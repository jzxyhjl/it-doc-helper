# Cursor 规则配置说明

## 什么是 .cursorrules 文件？

`.cursorrules` 是 Cursor IDE 的配置文件，用于定义 AI 助手的行为规则和开发规范。当你在项目中使用 Cursor 时，AI 助手会自动读取并遵循这些规则。

## 配置文件位置

配置文件位于项目根目录：`.cursorrules`

```
it-doc-helper/
├── .cursorrules          # Cursor 规则配置文件
├── README.md
└── ...
```

## 配置方法

### 方法一：自动配置（推荐）

项目已经包含了 `.cursorrules` 文件，Cursor 会自动识别并使用。

### 方法二：手动配置

如果项目中没有 `.cursorrules` 文件，可以手动创建：

1. **在项目根目录创建文件**
```bash
cd /Users/ggsk/Cursor/it-doc-helper
touch .cursorrules
```

2. **复制规则内容**
将规则内容复制到 `.cursorrules` 文件中，或参考项目中的 `.cursorrules` 文件。

3. **保存文件**
保存后，Cursor 会自动加载规则。

## 规则文件说明

### 核心规则

`.cursorrules` 文件包含以下主要规则：

1. **核心行为规则**
   - 使用中文回答
   - 主动审视需求
   - 最小可用实现原则

2. **技术栈要求**
   - Docker 容器化部署
   - 后端：Java (Spring Boot) 或 Python3 (Flask/FastAPI)
   - 前端：Vue.js（注意：当前项目使用 React）
   - 版本控制：Git

3. **工作流程**
   - 智能任务分级
   - Spec 工作流程
   - 变更记录规范

4. **输出风格**
   - 简洁直接
   - 可执行建议
   - 基于事实

### 注意事项

⚠️ **重要提示：**

1. **技术栈差异**
   - 规则中提到前端使用 Vue.js，但当前项目实际使用 **React**
   - 规则中提到后端使用 Flask，但当前项目实际使用 **FastAPI**
   - AI 助手会根据项目实际情况自动适配，但建议根据项目需求调整规则

2. **规则优先级**
   - `.cursorrules` 文件中的规则优先级最高
   - 如果规则与项目实际情况冲突，AI 会优先遵循项目实际情况

3. **规则更新**
   - 修改 `.cursorrules` 文件后，Cursor 会自动重新加载
   - 无需重启 Cursor

3. **检查行为**

   - AI 应该主动审视需求

## 自定义规则

### 如何修改规则

1. **编辑 `.cursorrules` 文件**
```bash
# 使用任意编辑器打开
code .cursorrules
# 或
vim .cursorrules
```

2. **修改规则内容**
根据项目需求修改规则，例如：
- 修改技术栈要求
- 调整工作流程
- 添加项目特定规则

3. **保存文件**
保存后，Cursor 会自动重新加载规则。

### 规则格式

`.cursorrules` 文件使用 Markdown 格式，支持：
- 标题（#、##、###）
- 列表（-、1.）
- 代码块（```）
- 强调（**、*）

### 示例：添加项目特定规则

```markdown
## 项目特定规则

- 所有 API 接口必须包含 Swagger 文档
- 前端组件必须使用 TypeScript
- 数据库迁移必须使用 Alembic
```

## 常见问题

### Q: 规则文件不生效？

**A:** 检查以下几点：
1. 文件是否在项目根目录
2. 文件名是否为 `.cursorrules`（注意前面的点）
3. 文件格式是否正确（Markdown）
4. 重启 Cursor 试试

### Q: 如何禁用某些规则？

**A:** 在 `.cursorrules` 文件中注释掉不需要的规则，或直接删除。

### Q: 规则与项目冲突怎么办？

**A:** AI 助手会优先遵循项目实际情况。如果规则与项目冲突，建议：
1. 根据项目实际情况修改规则
2. 或在规则中添加说明，让 AI 知道项目使用的实际技术栈

### Q: 多个项目可以共享规则吗？

**A:** 可以，但建议每个项目有自己的 `.cursorrules` 文件，因为不同项目可能有不同的规范。

## 最佳实践

1. **保持规则简洁**
   - 只包含必要的规则
   - 避免过于复杂的规则

2. **定期更新规则**
   - 随着项目发展更新规则
   - 移除不再适用的规则

3. **团队协作**
   - 将 `.cursorrules` 文件提交到 Git
   - 团队成员共享相同的规则

4. **文档化**
   - 在规则文件中添加注释说明
   - 解释为什么需要某些规则
